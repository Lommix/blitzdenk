Architecture

Architecture

Startup flow

  1. main.zig splits CLI flags and positional commands.
  2. Config files are created when missing.
  3. ContextFactory loads default tools, agents, modes, and available skills.
  4. The terminal and App are initialized.
  5. A Swarm is created and wired to app callbacks for broadcasts and permissions.
  6. Lua config is loaded, then project-local Lua is loaded when present.
  7. The TUI loop ticks the app, agents, tools, providers, MCP, and LSP.

Tool execution flow

  1. A provider returns tool calls in a response or stream.
  2. Agent.tick starts and tracks tool calls.
  3. The tool receives a ToolContext and ToolCall.
  4. The tool may request permission, update visible status, run commands, spawn child agents, or return immediately.
  5. The result is appended to chat and streamed back into the provider loop.

Extension flow

Lua calls into ContextFactory and the app:

  • register a provider with blitz.add_provider
  • set models with blitz.set_model or blitz.set_model_agent
  • register tools with blitz.register_tool
  • attach tools to agents with blitz.add_tool or blitz.set_agent_tools
  • add agents with blitz.add_agent
  • add modes with blitz.add_mode
  • register commands, key binds, event listeners, MCP servers, and LSP servers

The factory resets Lua-installed definitions on reload and keeps embedded defaults available.
